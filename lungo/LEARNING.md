# Learning

## Overview
- `aml314b/` is the AML 314(b) lane for lungo. It is separate from the coffee auction and logistics demos and now covers discovery, collaboration, and Phase 3.d investigation-type transport segmentation.
- `README_aml.md` at the `coffee_agents/lungo/` root is the operator-facing runbook for the AML lane. It now documents both seeded investigation categories, the `A2A` compatibility path, the `SLIM` lane-routing path, and the focused Phase 3.d validation command.
- `docker-compose.aml314b.yaml` is the minimal Docker path for the AML SLIM lane. It starts only `slim`, `fi_a`, and `fi_b` through `fi_f`, using `docker/Dockerfile.aml314b` instead of the larger auction/logistics compose stack.
- In `docker-compose.aml314b.yaml`, the `slim` gateway stays internal to the compose network. `FI_A` talks to it through `http://slim:46357`, and the host port is intentionally not published so the AML stack does not collide with other local SLIM instances.
- The AML-only compose path sets `OTEL_SDK_DISABLED=true` because `AgntcyFactory(..., enable_tracing=True)` otherwise tries to export to `localhost:4318`. That keeps the minimal stack independent from the observability containers.
- `docker/Dockerfile.aml314b` is the shared Python image recipe for the AML containers. It installs `uv`, creates `/venv`, pins `uv` to that virtualenv with `VIRTUAL_ENV` plus `UV_PROJECT_ENVIRONMENT`, installs dependencies from `pyproject.toml` and `uv.lock`, and then copies in the `lungo` source tree.
- The AML Docker setup needs a compose-specific directory file at `aml314b/data/counterparty_directory.docker.csv` because container-to-container traffic must use Docker service names instead of `127.0.0.1`. In particular, the SLIM endpoint must be `http://slim:46357`, not `http://localhost:46357`.
- The minimal AML compose path is intentionally backend-only. It starts `slim`, `aml-fi-a`, and `aml-fi-b` through `aml-fi-f`, exposes only `FI_A` on host port `9110`, and leaves the AML frontend to be run separately.
- `lungo/docker-compose.yaml` hard-codes `container_name` for every AML service, including `lungo-aml314b-slim` and `lungo-aml314b-fi-b` through `lungo-aml314b-fi-f`. That means the local AML compose path supports only one live stack at a time; a second `docker compose up --build` from another session will hit daemon name conflicts until the first stack is removed with `docker compose down` or the stale containers are deleted.
- `aml314b/frontend/` is the standalone AML browser app. It is a separate Vite package with its own `package.json`, `src/App.tsx`, `src/main.tsx`, and `src/styles.css`, and it uses `VITE_AML_API_BASE_URL` to talk directly to `fi_a`.
- `aml314b/common/` contains the shared schemas, collaboration helpers, enforcement hooks, step-event buffer, and CSV-backed stores reused across the AML discovery and collaboration flows.
- `aml314b/common/probing.py` now owns the Phase 3.e lane-probe contract and transport helper layer. It defines `LaneProbeRequest`, `LaneProbeResponse`, per-lane NATS subjects such as `aml314b.probe.money_mule`, a bounded `LaneProbeNatsClient`, and the responder-side `LaneProbeResponderRuntime`.
- `aml314b/fi_a/` contains the AML supervisor FastAPI app, the prompt parsing graph, and the orchestration tools that now chain discovery into collaboration for the new API path.
- `aml314b/institutions/fi_b` through `fi_f` remain thin responder wrappers with per-institution data and cards, while `aml314b/institutions/common/` now handles both discovery and collaboration payloads.
- `PHASE3D_EXECPLAN.md` in `coffee_agents/lungo/` is the checked-in execution plan for Phase 3.d investigation-type transport segmentation. The implemented path now uses explicit `investigation_type` fields everywhere, shared lane resolution in `aml314b/common/channeling.py`, lane-aware SLIM topics, and direct `A2A` as a simpler explicit-type compatibility path.
- `PROJECTBRIEF.md` now defines Phase 3.e as category narrowing before case-scoped discovery, and moves the earlier policy-and-disclosure expansion to Phase 3.f.
- `PHASE3E_EXECPLAN.md` now locks the clarified design: `FI_A` does not know lane membership ahead of time, responder institutions decide locally via compliance-configured lane subscriptions, the lane probe uses `NATS` pub-sub, only subscribed institutions reply `YES`, and only then does the existing explicit AML case discovery flow start. The probe stage is explicitly not a group session.
- `docker-compose.yaml` now includes an internal `nats` service and sets `AML314B_PROBE_NATS_ENDPOINT=nats://nats:4222` for every AML container so the Phase 3.e probe path works inside the compose network instead of relying on a host-local NATS process.

## Discovery Flow
- `aml314b/fi_a/graph/graph.py` now resolves `investigation_type` from either the explicit request field or prompt text and rejects mismatches deterministically before orchestration starts.
- `aml314b/fi_a/graph/tools.py` is the orchestration core. It loads FI_A's active case and directory stores, resolves the investigation lane, sends one `DiscoveryRequest` per responder, and aggregates five `DiscoveryResponse` objects into a `DiscoveryAggregateResult` that now includes `investigation_type`, `transport_lane`, and `discovery_session_id`.
- `aml314b/fi_a/graph/tools.py` now resolves explicit discovery targets through `discover_lane_candidates(...)` before sending case-scoped discovery. The aggregate result includes `candidate_institutions`, `candidate_response_count`, and `candidate_resolution_source="NATS_LANE_PROBE"`, while new step events (`lane_probe_sent`, `lane_probe_response_received`, `candidate_set_finalized`) keep the probe stage separate from later discovery and collaboration events.
- The deterministic text contract is: `Discovery completed for case <case_id> and entity <entity_name> (<entity_id>). Accepted: <...>. Declined: <...>.`

## Collaboration Flow
- `aml314b/common/collaboration.py` builds and validates `CollaborationSessionRequest` objects. The participant list must be exactly `FI_A` plus the affirmative discovery responders, in deterministic order.
- `aml314b/institutions/common/collaboration_agent.py` loads one bounded statement from each institution's `curated_investigative_context.csv` file, validates the investigation metadata, and rejects institutions that were not selected into the collaboration session.
- Each responder data directory now includes `lane_subscriptions.csv`. `LaneSubscriptionStore` loads that file, `runtime.py` uses it to decide both which NATS probe subjects to subscribe to and which SLIM lane registrations to expose, and the responder agents reject investigation types that are not locally subscribed.
- `aml314b/fi_a/graph/tools.py` exposes `run_discovery_and_collaboration(...)` and `run_collaboration_from_discovery(...)`. The implementation creates a fresh `collaboration-<uuid>` session, requests one contribution from each accepted institution, and threads `investigation_type` plus optional `transport_lane` through the result objects.
- `aml314b/fi_a/main.py` now exposes both compatibility prompt endpoints and the new structured AML endpoints: `POST /agent/probe`, `GET /agent/cases?investigation_type=...`, `POST /agent/cases/run`, and `GET /agent/step-events`. The structured path lets the frontend probe a lane, list active cases from `ActiveInvestigationsStore`, and run discovery or collaboration from `case_id` without prompt parsing.

## Real-World Channel Design
- In the current demo, transport-channel entry is controlled mostly by configuration, directory routing, topic/session registration, and collaboration participant selection, not by a strong authorization layer.
- A production design should treat channel entry as an identity-plus-policy problem: each institution service should have a strong workload identity, and the transport broker should enforce authenticated publish/subscribe or session-join permissions.
- Investigation-type isolation should be modeled in two layers: a coarse lane such as `money_mule` or `terrorist_financing`, plus a case-scoped session inside that lane for a specific investigation.
- For a real deployment, the control plane should mint short-lived credentials bound to institution identity, investigation type, case ID, and role. The broker should enforce those credentials, and the receiving agents should also validate message claims locally.
- Auditability should cover channel creation, join decisions, publishes, receives, and policy outcomes so a reviewer can reconstruct exactly who entered which lane and why.
